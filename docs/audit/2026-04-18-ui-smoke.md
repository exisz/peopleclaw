### Bug 12 (MINOR): Login flow forces "Create passkey" and "Enable 2FA" interstitials with no in-flow opt-out hint
- **Flow:** login-flow
- **Severity:** minor (acquired-on-the-way-in friction)
- **Steps:**

  2. Logto presents `/create-passkey` with only one CTA "Create a passkey" (Skip is in the top-right nav as a small div, easy to miss).

- **Suggested ticket title:** Logto post-signin onboarding (passkey, MFA) needs prominent "Maybe later" or be config-disabled

## Recommendations to unblock future automated audits
1. **Seed a deterministic demo tenant + workflow** (Bug 1). The simplest path: a `pnpm seed:e2e` script that ensures `demo_acceptance_test` user exists, has a tenant `acceptance`, with workflow `shopify-auto-smoketest` containing 3 nodes; idempotent.
2. **Add `/api/test/sudo-login`** behind a `E2E_TEST_TOKEN` env flag that mints a session for a given user id, bypassing Logto entirely. Cuts our auth setup from ~90 lines of interstitial-juggling to one HTTP call.
3. **Disable Logto post-signin passkey + MFA-onboarding prompts globally** (Sign-in Experience → MFA `policy: NoPrompt`, and remove WebAuthn from required factors). They're optional anyway; the prompt friction is acquired-on-the-way-in cost.
4. **Add `[role="tab"]`/`[role="tablist"]` to the editor's right-panel tabs** — improves both a11y and test stability (see Bug 7).
5. **Stamp Step Library items with `data-testid="step-template-{slug}"`** so future drag-tests can locate them deterministically.
